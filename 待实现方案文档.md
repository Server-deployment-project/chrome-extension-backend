# RAG知识库基于PGVector实现方案

## 一、 核心技术栈与依赖
1. **向量数据库**: PostgreSQL + PGVector 插件。
2. **Embedding 模型**: OpenAI `text-embedding-ada-002` 或 `text-embedding-3-small` (1536维)。
3. **框架支持**: 引入 `spring-ai-pgvector-store-spring-boot-starter` 和 `spring-ai-openai-spring-boot-starter`。

## 二、 架构与模块设计

### 1. 数据库配置 (PGVector)
在 PostgreSQL 中启用 `vector` 扩展，并配置包含 1536 维向量列的数据表，使用余弦相似度（Cosine Similarity）作为默认距离度量。

### 2. 文档预处理与分块策略
- **数据清洗**: 去除无效空白符、特殊不可见字符等。
- **TokenTextSplitter**: 按照指定的 Token 数量（如 500 tokens/chunk，重叠 50 tokens）对文本进行分割，保留上下文语义连贯性。
- **元数据 (Metadata)**: 在处理每个文本块（Chunk）时，注入预置的元数据标签（如：标尺、类别、文档名称、上传时间等），用于后续高效的过滤（Filter）。

### 3. 核心接口划分

**A. 文档上传与向量化入库接口**
- **Endpoint**: `POST /api/knowledge-base/upload`
- **流程**:
  1. 接收用户上传的文档（支持 txt, pdf, markdown 等格式）及相关分类/标签参数。
  2. 提取文本，进行清洗标准化。
  3. 使用 `TokenTextSplitter` 进行切片。
  4. 为每个切片（Document 实例）注入元数据标签。
  5. 批量调用 OpenAI Embedding 接口生成向量。
  6. 存入 PGVector 数据库。

**B. 知识库检索接口**
- **Endpoint**: `POST /api/knowledge-base/search`
- **流程**:
  1. 接收用户的查询词（Query）、`topK`（返回数量限制）、`filterExpression`（过滤条件，如标签匹配等）。
  2. 构建 Spring AI 的 `SearchRequest` 构建器，设置 `query`、`topK` 和 `filterExpression`。
  3. 执行查询，返回匹配的文本块及其相似度分数和元数据。

## 三、 代码层级规划
- **Config 层**: 增加 `VectorStoreConfig` 配置 PGVector 连接与初始化，配置 OpenAI Embedding 模型参数。
- **Service 层**: 创建 `KnowledgeBaseService`，封装 Document 读取、`TokenTextSplitter` 逻辑和 `SearchRequest` 检索逻辑。
- **Controller 层**: 创建 `KnowledgeBaseController` 暴露 RESTful 接口。
- **DTO 层**: 增加 `KnowledgeUploadRequest` 和 `KnowledgeSearchRequest` 等数据载体。
